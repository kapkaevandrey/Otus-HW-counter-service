# Отчёт об архитектуре — сервис счётчиков непрочитанных сообщений

Домашнее задание: **сервис счётчиков** (вариант 1). Счётчик непрочитанных сообщений в диалогах. Ключевые задачи: отказоустойчивость, использование кеша, консистентность счётчика с реальным числом сообщений (паттерн **SAGA**), готовность к высокой нагрузке на чтение.

---

## 1. Контекст: три сервиса

| Сервис | Роль | Хранилище |
|--------|------|-----------|
| **Монолит** (`Otus-HW_01`) | пользователи, посты, прокси API диалогов | PostgreSQL (Citus) |
| **chat-service** (`Otus-HW-chats`) | диалоги, отправка/чтение сообщений | Redis (сообщения) + PostgreSQL (inbox пользователей) |
| **counter-service** (этот репозиторий) | счётчики непрочитанных, API чтения | Redis (Valkey) |

Счётчики вынесены в **отдельный сервис**, потому что у них другой профиль нагрузки (очень много чтений, простые операции) и другое хранилище (Redis-кеш), чем у диалогов. Это позволяет масштабировать чтение счётчиков независимо от chat-service.

---

## 2. Выбранный паттерн консистентности: choreography SAGA

Счётчик — это **производное состояние** от событий в диалогах. Синхронно обновлять его из chat-service нельзя: это связало бы два сервиса в распределённую транзакцию и уронило бы доступность. Поэтому используется **SAGA в стиле хореографии** поверх Kafka:

1. chat-service в рамках **своей** локальной транзакции меняет диалог **и** записывает событие в outbox (Redis Stream) — атомарно, одним Lua-скриптом.
2. Отдельный **relay-worker** chat-service читает outbox и публикует событие в Kafka.
3. counter-service — **consumer** этого топика — применяет событие к счётчику в Redis.

Никакого оркестратора: каждый сервис реагирует на события и делает свой локальный шаг. Итог — **eventual consistency**: счётчик сходится к реальному числу непрочитанных с задержкой в доли секунды.

### События SAGA

| Событие | Когда | Эффект на счётчик |
|---------|-------|-------------------|
| `message.sent` | A отправил сообщение B | `unread:{B}[A] += 1` |
| `dialog.read` | B открыл диалог с A | `unread:{B}[A] = 0` |

## 4. Модель данных в Redis (counter-service)

| Ключ | Тип | Содержимое |
|------|-----|------------|
| `unread:{user_id}` | HASH | поле = `peer_id`, значение = число непрочитанных от этого собеседника |
| `processed:{event_id}` | STRING (TTL) | маркер идемпотентности обработанного события |

Чтение счётчика:
- суммарно — `HGETALL unread:{user}` + сумма (или отдельный лёгкий эндпоинт `/total`);
- по собеседнику — `HGET`.

Это **кеш-ориентированная** модель: все операции O(1)/O(размер диалогов пользователя), без обращений к диску и без джойнов — то, что нужно под большую нагрузку на чтение.

---

## 5. Отказоустойчивость и корректность

### 5.1 Transactional outbox (chat-service)
Событие пишется в Redis Stream **в той же атомарной Lua-операции**, что и изменение диалога. Значит, не бывает ситуации «сообщение записано, а событие потеряно» и наоборот.

### 5.2 At-least-once + идемпотентность
Kafka и relay дают доставку **как минимум один раз** (возможны дубли при ретраях/reclaim). Идемпотентность обеспечивается на двух уровнях:
- **relay** (chat): `relay:sent:{event_id}` — не публиковать один и тот же event дважды;
- **consumer** (counter): Lua-скрипт делает `SET processed:{event_id} NX EX ttl` и меняет счётчик **только если ключ поставлен**. Повторная доставка того же `event_id` — no-op.

### 5.3 Атомарность операции счётчика
Проверка идемпотентности и мутация счётчика — в **одном Lua-скрипте** (Redis выполняет его атомарно, однопоточно). Нет окна между «проверил, что не обработано» и «увеличил».

- `increment_unread_on_message_sent.lua`: `SET NX` + `HINCRBY`
- `reset_unread_on_dialog_read.lua`: `SET NX` + `HSET 0`

### 5.4 Надёжное потребление
- consumer коммитит offset **вручную** после успешной обработки (`enable_auto_commit=false`);
- воркеры обёрнуты в `restart_on_exception` — сервис самовосстанавливается после сбоев;
- relay использует `XAUTOCLAIM` — «зависшие» (не подтверждённые упавшим воркером) сообщения переклеймливаются и допубликовываются.

### 5.5 Порядок событий и гонки — Kafka partition key
Один и тот же счётчик `unread:{B}[A]` меняют и `message.sent` (A→B), и `dialog.read` (B прочитал A). Если обрабатывать их параллельно, возможна **гонка** (инкремент и обнуление в неверном порядке → неверное значение).

Решение: **ключ партиционирования Kafka = владелец счётчика** (тот, чей `unread` меняется):

| Событие | Поле-владелец | Partition key |
|---------|---------------|---------------|
| `message.sent` | `recipient_id` | `str(recipient_id)` |
| `dialog.read` | `user_id` | `str(user_id)` |

Для сценария A→B оба события дают **один и тот же UUID (B)** → одна партиция → строго последовательная обработка в одном consumer → гонки нет.

Почему именно владелец счётчика, а не `conversation_id`: ключ совпадает с **агрегатом, чьё состояние мутируется** (`unread:{user}`). Это корректно и в 1-на-1, и обобщается на групповые чаты / «прочитать всё», не создавая горячих партиций на активный диалог. (Подробности — в коде relay, метод `_counter_owner_id`.)

## 6. Масштабирование под нагрузку на чтение

- **Чтение** обслуживается из Redis (кеш) за O(1) — отдельно от chat-service; можно поднять несколько реплик API (`app-instance` в compose).
- **Запись** (обработка событий) масштабируется числом партиций топика и инстансов consumer в одной consumer-group; порядок в пределах счётчика сохраняется благодаря partition key.
- Чтение и запись физически разделены (CQRS-подобно): тяжёлое чтение не конкурирует с обработкой событий.

## 8. Соответствие критериям ДЗ

| Требование ДЗ | Реализация |
|---------------|-----------|
| Сервис счётчиков непрочитанных | отдельный сервис, Redis `unread:{user}`, consumer + API |
| Большая нагрузка на чтение | Redis-кеш, чтение O(1), отдельный `/total`, реплики API, разделение чтения/записи |
| Использование кешей | Redis/Valkey как основное хранилище счётчиков |
| Консистентность (SAGA) | choreography SAGA: outbox → Kafka → consumer, eventual consistency |
| Отказоустойчивость | transactional outbox, at-least-once + идемпотентность, ручной commit, `XAUTOCLAIM`, авто-рестарт воркеров |
| Корректность при гонках | partition key по владельцу счётчика + атомарные Lua-скрипты |
